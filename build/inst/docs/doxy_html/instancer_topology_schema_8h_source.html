58 // Since the instancing schema is complicated:
59 //
60 // An instancer is a prim at a certain scenegraph location that causes other
61 // prims to be duplicated. The instancer can also hold instance-varying data
62 // like constant primvars or material relationships.
63 //
64 // The important things an instancer has is:
65 // 1.) Instancer topology, describing how exactly the prims are duplicated;
66 // 2.) Instance-rate data, meaning data that varies per instance, such as
67 // primvars or material bindings.
68 //
69 // If an instancer causes prims "/A" and "/B" to be duplicated, we encode that
70 // by setting prototypes = ["/A", "/B"]. Note that "/A" and "/B" can be
71 // subtrees, not direct gprims. instanceIndices encodes both multiplicity
72 // and position in arrays of instance-rate data, per prototype path; if
73 // instanceIndices = { [0,2], [1] }, then we draw /A twice (with instance
74 // primvar indices 0 and 2); and /B once (with instance primvar index 1).
75 // Mask is an auxiliary parameter that can be used to deactivate certain
76 // instances; mask = [true, true, false] would disable the
77 // second copy of "/A". An empty mask array is the same as all-true.
78 //
79 // Scenes generally specify instancing in one of two ways:
80 // 1.) Explicit instancing: prim /Instancer wants to draw its subtree at
81 // an array of locations. This is a data expansion form.
82 // 2.) Implicit instancing: prims /X and /Y are marked as being identical,
83 // and scene load replaces them with a single prim and an instancer.
84 // This is a data coalescing form.
85 //
86 // For implicit instancing, we want to know the original paths of /X and /Y,
87 // for doing things like resolving inheritance. This is encoded in the
88 // "instanceLocations" path, while the prototype prims (e.g. /_Prototype/Cube,
89 // the deduplicated version of /X/Cube and /Y/Cube) is encoded in the
90 // "prototypes" path.
91 //
92 // For explicit instancing, the "instanceLocations" attribute is meaningless
93 // and should be left null.
